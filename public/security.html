Railway Security

A Personal Health Train implementation

Medical Data Works's Railway service implements a privacy preserving federated infrastructure as proposed by the Personal Health Train. The main principle of the Personal Health Train is to bring questions to the data rather than moving data. This concept is called Federated Learning.

In this section, the security of Railway is discussed.

Definitions

The Personal Health Train metaphor defines the following components:

  • Station: An information system that contains sensitive data, such as from patients. In vantage6, this corresponds to node hosted by an organization as part of a collaboration. Typically a hospital or registry provides a Station.
  • Train: Software that encodes a question to be asked to a Station and outputs anonymous, statistical data. Typically a university or a company provides a Train.
  • Track: Software and infrastructure that allow Trains to enter Stations and ask questions. Medical Data Works B.V. provides the Track.

Security Practices

WORK IN PROGRESS

Medical Data Works has setup a security framework for Railway projects adhering to the following practices:

  • Tracks are TLS protected. We use an https endpoint for our vantage6 server. This protects users form eavesdropping external attackers and Man-in-the-middle attacks.
  • We use a separate vantage6 servers for different projects housed in our infrastructure, furthering compartmentalization. This way, a potentially compromised Railway project need not imply that a different Railway project is also compromised.
Support for vantage6 security features
We offer support for some security settings which vantage6 implements. With them, stations can further enhance their security by:
  • Only allowing a specific set of vetted trains to enter their station. This is, only allowed algorithms will run on a node (station). See vantage6 documentation here. This could prevent an organization from running a malicious algorithms that hasn't met their explicit approval.
  • Only allowing a specific set of train providers to enter their station. Thereby, for example, only allowing certain universities and institutes to pose questions to the data in a station.
  • Only allowing trains that come from a specific railway yard (docker registry). An organization might want to place trust in a single docker registry where only a specific set of users have upload access.
  • Additionally, communications between nodes and users can be encrypted so as to decrease the server's visibility into those communications. See pertinent vantage6 documentation here.
